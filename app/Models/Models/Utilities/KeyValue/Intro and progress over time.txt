OEM generally define their vehicle in Model having variant having color and the specificaions like (Segment, Sub Segment, Colour Code, Colour, Fuel, GST, Seating, Wheels, Transmission, Drivetrain, Body Make, Body Type, CC, CC Range, GVW, Permit, Taxi Price, Status) are defined at variant level along-with exshoroom price. Colors are generally defined by types (Each brand use its own color type) like General, Matelic, Dual-Tone, Spl. and some times these specific color comes at a premium increasing the exshowroom price for a specific color. Every OEM-Brand give a color code and a name to color at VehicleModel level. Some time same color code have differnt name/meaning for different VehicleModle i.e. SB may be sky blue for one model and Silver-Black for other. Every Varinat is given a unique Code generally known as OEM-code and it comes with a suffix of its color code so if a vehicle of code abcdef19 is coming in 5 color (wd,tl,gr,bl,on) it will have 5 entires abcdef19wd, abcdef19tl, abcdef19gr, abcdef19bl and abcdef19on but we create just one Variat and the 5 entries of varint_color for all of its colors. Every Brand has a specific char of color code like Mahindra use only last 2 chars or OEM-code but Honda use last 3 chars of OEM-Code for colors (In Honda fist 2 char of color are for Body color while last 1 is for interior color).
Variant level vehicle specification are generally sotred via our Keyword/KeyValue based Status/Enum mechanism as they repeats for vehicles and we can search/query them based on these values like Fuel has fix values to choose from Petrol/Diesel/CNG/Electric or Permit have Passanger/Private/Goods/Misc allowed values.
For the ease of query and avoid joins we normalise our tabels in such a way that each entity will have direct link to all of its ancestores lika a variant will have VehicleModel_id, Sub-Segment_id (if apllies else null), Segment_id and Brand_id; while the color will have variant_id to apart from these values. <Approval Flow>
In daily business process like sales different kinds of special approvals (generally for pricing) are needed and these appravals may go upto 7-8 level up or sometime finish at only two levels it depends on the size of dealership and volume of sales it is handling and totally customisable for every case. Now at each level there are different user having different approval power/limit (this needs to be manage by JSON column as differen flows use different structrue to define this power. Now the creator of sales, quote, Booking needs someones approval before giving custom values or special discount/price or rebate. So he will create an approval case as per the workflow by default this case is assigned to lowest level approver, now this approver check if the ask is within his power/limit or not and what to do with it? If the ask is in his limit then he has the power to accept the ask in full, or give counter offer (Bargain) or reject the ask outrightly; now if ask is beyond his power he still can bargain or reject or esclate it to next level (if there is a next level approver available). So every approver do the same and the esclation loop will continue till it comes to the highest level of approver. We have to keep track of each approvers action and modificaiton to the original ask via json columns as the structure will be different for different topics, but procdeure will be the same of apply, modify, esclate or reject. A modified or accepted or rejected case will goes back directly to the source and each approver and creator will be able to track each case gone through him.
See sample workbook Reproting-N-Approval.xlsx for details
<Approval Flow>
